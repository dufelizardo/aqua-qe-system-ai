Score: 65 · risk: high
AMB-001critical
Lockout threshold not specified in RN-07
RN-07 states the system must block after "X consecutive invalid attempts" but the value of X is not defined. This makes the rule untestable and unimplementable.
→ Define the exact threshold (e.g. 5 attempts) and lockout duration (e.g. 15 minutes) in RN-07.
AMB-002high
Session timeout duration undefined in RN-09
RN-09 mentions "inactivity timeout" but does not specify the duration. Without this, QA cannot test the session expiration behavior.
→ Specify timeout value (e.g. 30 minutes of inactivity).
AMB-003medium
"Temporary" block duration not defined (RN-07)
The block is described as "temporary" but no duration or unblock mechanism is specified. What happens after the block? Auto-unblock? Admin intervention?
→ Define block duration and unblock mechanism.
AMB-004medium
Password recovery flow scope unclear (RN-08)
RN-08 says the system must "provide" the forgot-password feature but does not define the recovery method (email link, SMS code, security questions).
→ Specify recovery mechanism in RN-08.
RSK-001critical
Brute force attack vector — undefined lockout
Without a specified lockout threshold in RN-07, the login endpoint is vulnerable to brute force attacks. An attacker can try unlimited password combinations.
→ Implement rate limiting + lockout with defined threshold before go-live.
RSK-002high
Timing attack on credential validation
If the system returns different response times for "user not found" vs "wrong password", attackers can enumerate valid usernames.
→ Ensure constant-time response for all invalid credential scenarios (NFR-001).
RSK-003high
Session not invalidated server-side on logout
If only the client-side token is cleared without server-side invalidation (RN-09), stolen tokens remain valid after logout.
→ Implement server-side session revocation on logout.
RSK-004high
No LGPD audit logging specified
Failed login attempts, lockout events, and password recovery requests must be logged for LGPD compliance. None of the RNs mention audit logging.
→ Add audit logging requirement (NFR-003).
RSK-005medium
No MFA requirement for sensitive roles
The requirement does not address Multi-Factor Authentication. For an educational platform with student data, MFA may be required for admin roles.
→ Evaluate MFA requirement for privileged profiles.

## Business Value — EDUIA-1000 Login Screen

### What this delivers to users
Students and educators gain secure, reliable access to the EDUIA platform.
The login screen is the gateway to all platform functionality — getting it right
is foundational to all other features.

### What changes in the operation
- Users authenticate with email + password (replacing any current workaround)
- Session management prevents users from needing to re-login constantly
- Password recovery reduces support tickets for forgotten credentials

### Risks to the business (translated from technical findings)

⚠ **CRITICAL — Brute force vulnerability (RSK-001)**
Without a defined lockout threshold (RN-07 is incomplete), student accounts
can be compromised through automated password guessing. This could expose
student personal data and create LGPD liability.
→ Action needed: Define lockout threshold before go-live.

⚠ **HIGH — LGPD compliance gap (RSK-004)**
Authentication events are not being logged. Brazilian LGPD requires audit
trails for data access events. Non-compliance risks regulatory fines.
→ Action needed: Add audit logging requirement.

### What users will NOT be able to do (gaps identified)
✗ No password complexity requirements specified — weak passwords allowed
✗ Password recovery flow exists (RN-08) but has no testable acceptance criterion
✗ No "remember me" functionality specified
✗ No session timeout visible to the user (silent expiration)

### Recommended priority before go-live
| Item | Business Impact | Effort | Priority |
|------|----------------|--------|----------|
| Define lockout threshold (RN-07) | Critical — security | Low (spec only) | P0 |
| Add CA for password recovery (CA-07) | High — testability | Low (spec only) | P0 |
| Add LGPD audit logging | Critical — compliance | Medium | P1 |
| Define session timeout value (RN-09) | High — UX + security | Low | P1 |

### How to measure success
- Login success rate > 98% in first 30 days
- Support tickets for "can't login" < 5% of user base
- Zero security incidents related to credential attacks
- Zero LGPD violations related to authentication events
