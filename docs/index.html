What is the AQuA Intelligence Platform?

The AQuA Intelligence Platform is an internal platform that analyzes software requirements using a multi-engine AI pipeline. Each requirement is processed by specialized engines that run in parallel, then a Synthesis Engine correlates all findings to produce actionable insights.

The system learns from every analysis — building a corporate Knowledge Layer that grows smarter over time, allowing it to detect patterns, flag recurring risks, and suggest test coverage based on your organization's history.

Who is it for?

🧪
QA Engineers
Get auto-generated test scenarios, RTM, Gherkin files, and evidence templates based on real business rules and acceptance criteria.
💻
Developers
Receive technical gap analysis, automation skeletons (Robot Framework, Playwright, pytest), and identified missing specifications.
📋
Product Owners
See inferred acceptance criteria, functional gaps translated into user impact, and business value reports in plain language.
🔐
Audit & Compliance
Access full Risk Reports with historical correlation, exportable RTMs, and regulatory compliance flags (LGPD, ISO standards).

Docs / Core Concepts

Core Concepts

Key ideas you need to understand before working with the system.

Story ID Detection

The system automatically detects story IDs in the format ABC-1234 (e.g. EDUIA-1724) from requirement text. When detected, all analysis results are versioned and linked to that story ID.

💡
Paste the full user story text including the story ID. The system extracts it automatically and links versions, defects, and KB entries to that story.

RNs and CAs

The Requirement Parser extracts structured items from requirement text:

NotationMeaningExample
RN-01Business Rule (Regra de Negócio)RN-01: The card must be displayed alphabetically
CA-01Acceptance Criterion (Critério de Aceite)CA-01: A card with title "TEXT Query" must exist
RF-01Functional RequirementInferred by the Inference Engine
RNF-01Non-Functional RequirementInferred by the Inference Engine

Versioning

Every time you analyze the same story ID, the system creates a new version. Versions are immutable — v1 always reflects what was analyzed at that point in time.

In story.html, the Versions tab shows the full timeline with score evolution, delta between versions (what changed in RNs and CAs), and impact propagation (which test cases are affected).

Knowledge Base

The corporate Knowledge Base (KB) stores patterns, risks, bugs, compliance rules, and heuristics. It has two sources:

  • Manual entries — added via the API or seeded from seed_knowledge.py
  • Auto-extracted entries — added automatically after each analysis (badge: auto)

Confidence scores grow with repetition: a pattern seen once = 30% confidence; seen 8+ times = 90% confidence.

Stage 1 — Normalizer

The Normalizer is the first engine to run. It does two things simultaneously:

  • Uses AI to extract intent, actors, actions, conditions, and constraints from the raw requirement text
  • Uses the Requirement Parser (no AI needed) to extract every RN and CA with their exact description text

Corporate KB context is injected into the prompt before the AI call, so the Normalizer already knows about relevant historical patterns when it runs.

Stage 2 — Parallel Engines

Six engines run simultaneously after the Normalizer:

EngineOutputUsed by
AmbiguitySubjective terms, missing conditions, inconsistenciesQA, PO
RiskFunctional, security, regulatory, technical risks with severityQA, Audit
RulesExplicit and implicit business rules extractedQA, Dev
GapMissing scenarios, absent dependencies, incomplete flowsQA, PO, Dev
CoverageCovered vs uncovered acceptance criteriaQA
InferenceImplicit FRs and NFRs derived from RNs, CAs, risks, domainPO, Dev

Stage 2.5 — Knowledge Aggregator + Traceability

These two engines run in parallel after the 6 engines above but before Synthesis:

  • Knowledge Aggregator — fetches all historical data for this story: previous versions, defects, KB entries, family links. No AI call — pure data aggregation.
  • Traceability Engine — generates RTM, test scenarios, and test cases using the structured RNs and CAs from the parser combined with all engine findings.

Stage 3 — Synthesis

The Synthesis Engine receives all findings from every engine plus the historical context from the Knowledge Aggregator. It produces:

  • An overall quality score (0–100)
  • Cross-engine correlations — insights that no single engine could produce alone
  • A verdict: is this requirement ready for development and testing?
  • Top actions to address before proceeding
💡
Example correlation: "Risk RSK-003 (unauthorized access via URL) has no corresponding negative test case in the Coverage Engine output — CT covering RN-05 and CA-06 is critical before go-live."

Semantic Search

The system uses TF-IDF semantic search to find related KB entries by meaning — not just exact keyword matching.

✓
Searching for "payment timeout" finds "gateway failure" entries. Searching for "account lockout" finds "invalid attempt blocking".

Each engine receives a specialized context slice:

EngineReceives from KB
Risk EngineHistorical risks only (risco, comp)
Gap EngineRecurring gaps and patterns (padrao, heur)
SynthesisCorrelations and organizational context (heur, risco, comp)

Auto-enrichment

After every completed analysis, the Knowledge Enricher automatically extracts learnable patterns and saves them to the KB:

  • Critical/high risks → saved as risco entries
  • Gaps → saved as padrao entries
  • High-priority NF requirements → saved as comp or risco
  • Synthesis correlations → saved as heur

Duplicate detection prevents noise — if a similar entry already exists, the system increments its occurrence count instead of creating a new entry.

Confidence Scoring

OccurrencesConfidenceMeaning
130%First occurrence — low signal
360%Emerging pattern
575%Established pattern
8+90%+Strong organizational pattern — high priority in prompts

Version Delta & Impact

When a story is analyzed multiple times, the system computes the delta between versions:

  • Which RNs were added, removed, or modified?
  • Which CAs changed?
  • What is the overall impact level? (none / low / medium / high / critical)

The Impact Propagation then identifies which existing test cases and automations are affected by the detected changes — recommending re-execution or re-analysis.

Traceability & RTM

Full requirements traceability matrix generated automatically from your RNs and CAs.

How the RTM is generated

The Traceability Engine v2 uses the exact RN and CA texts extracted by the Requirement Parser — not AI-invented descriptions. Every test case references a real rule and a real criterion from your requirement.

Test Case Format

CT-001 - RN-01 | CA-01: Card "Consulta de TEXT" displayed on Manager Home
  RN-01: The functionality must be available via a card on the Manager Home
  CA-01: A card with the title "Consulta de TEXT" must exist

  Pre-conditions:
  - Authenticated user with CONSULTA_TEXT_ACCESS permission

  Test data:
  - User: [email protected] (valid permission)

  Steps:
  1. Navigate to Manager Home
  2. Verify the "Consulta de TEXT" card is visible
  3. Verify alphabetical ordering among other cards

  Expected result: Card is displayed in alphabetical position per RN-02

Scenario Coverage

The Traceability Engine enforces coverage rules:

✓

Docs / Requirements Inference

Requirements Inference

Discover implicit requirements that nobody wrote but everyone expects.

What it solves

A requirement may state: "RN-01: The system must block after 3 failed attempts."

But nobody wrote the implicit requirements that need to exist for this to work:

RF-001 [functional] The system must maintain an attempt counter per user session
RF-002 [functional] The system must provide an account unblocking mechanism
RNF-001 [security]  The block must persist across sessions (not reset on browser close)
RNF-002 [performance] Block decision must execute in under 200ms
RNF-003 [compliance] All blocked attempts must be written to the audit log (LGPD)

Output structure

FieldDescription
idRF-001, RNF-003 etc.
typeRF (functional) or RNF (non-functional)
categoryfuncional / performance / seguranca / disponibilidade / compliance / usabilidade
titleShort, actionable requirement title
descriptionFull description of what must be implemented
originWhich RN / CA / risk / gap originated this inference
rationaleWhy this was inferred — the reasoning chain
prioritycritical / high / medium / low
💡
Inferred requirements are visible in the RF/RNF tab of story.html and also appear in Artifact Generation — POs receive acceptance criteria derived from inferred FRs, and Audit receives NFRs in the Risk Report.

Generating artifacts

Artifacts are generated on-demand from an existing analysis:

  • In the Hub: open module 5 → select profile → click ⚡ Generate
  • In story.html: go to Artifacts tab → click ⚡ on any version

Generated artifacts are saved to the database and persist across sessions. Each artifact has a copy button and a download button with the correct extension (.feature, .robot, .md).

Switching AI providers

To switch from Gemini to Azure OpenAI — just update .env and restart:

# Before (Gemini)
AI_PROVIDER=gemini
AI_MODEL=gemini-2.0-flash
AI_API_KEY=AIza...

# After (Azure OpenAI)
AI_PROVIDER=b3gpt
AI_MODEL=gpt4o-mini
AI_API_KEY=your-token
BASE_URL=https://your-corporate-endpoint/openai
⚠
After changing providers, restart the server. The provider is cached as a singleton — changes only take effect on restart.

Tuning for large requirements

Requirements with 20+ RNs and 20+ CAs (like a full EDUIA story) need more tokens for the Traceability Engine to generate complete coverage:

ENGINE_MAX_TOKENS=8000   # minimum for large stories
ENGINE_TIMEOUT=180       # increase if traceability times out
Docs / FAQ & Troubleshooting

FAQ & Troubleshooting

Common issues

Server fails to start — circular import error

This happens when engines/__init__.py has imports. Keep it empty:

# engines/__init__.py — keep this empty
__all__ = ["normalizer", "ambiguity", "risk", "rules", "gap",
           "coverage", "synthesis", "knowledge_aggregator", "traceability"]

Import engines in orchestrator via import engines.X as X_mod — never from engines import X.

Analysis returns empty engine results

Check the error field in the response. Common causes:

  • API key invalid or expired → check AI_API_KEY in .env
  • Provider endpoint unreachable → check network / VPN
  • Token limit exceeded → increase ENGINE_MAX_TOKENS
  • Engine timeout → increase ENGINE_TIMEOUT

Traceability generates only 1-2 test cases

This is a token limit issue. The Traceability Engine needs more output space for large requirements:

ENGINE_MAX_TOKENS=8000

The engine automatically doubles its token allocation for traceability (max_tokens * 2).

Database schema error on startup

After modifying schemas.py or database.py, the existing database is incompatible:

del qa_system.db
py -3.12 seed_knowledge.py
⚠
This deletes all saved analyses and KB entries. Export or back up the database before deleting if you need to preserve data.

Frontend not loading (Illegal return statement)

This is a JavaScript syntax error caused by an orphaned code block outside a function. Usually happens after an edit that injects code incorrectly. Check the browser console for the line number, then verify that every return statement is inside a function.

Knowledge Layer shows 0 auto-extracted entries

The knowledge_enricher.py must be in backend/engines/. Also verify the analysis completed successfully — enrichment only runs after a successful pipeline execution.

Frequently asked questions

Can I use the system without a VPN / corporate network?

Yes, if you configure a public AI provider (Gemini or Anthropic). The B3GPT provider requires access to the corporate API endpoint.

Why does the system sometimes give different results for the same requirement?

LLMs are non-deterministic by nature. Temperature is set to 0.3 (low) to minimize variance, but some variation is expected. The structured elements (RN/CA extraction, RTM format) are deterministic — only the prose descriptions vary.

Can I add my own KB entries?

Yes — via the API:

POST /api/v1/knowledge
{
  "category": "risco",
  "title": "TEXT flows require mandatory audit trail",
  "description": "Any financial operation involving TEXT codes...",
  "project_id": "optional"
}

Or edit seed_knowledge.py and re-run it to populate the initial KB.

How do I export the RTM to Jira or Xray?

Currently, artifacts are exported as Markdown files. Xray and Jira direct export are on the roadmap. In the meantime, the Markdown RTM can be converted to CSV for import.
